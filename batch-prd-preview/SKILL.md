# Batch PRD Preview Skill

批量处理 Word 文件（.docx/.doc），转换为 Markdown 预览格式，执行需求评审，并生成汇总报告。

与 `prd2trd` skill 不同，本 skill **只做格式转换和需求评审**，不进入需求深挖、TRD规划、生成TRD等后续流程。

---

## 工作流程

```
批量扫描 → 格式转换 → 需求评审(Review) → 输出预览列表 → 生成汇总报告
```

#### 1.2 层级说明

| 层级           | 说明                                         | 示例                                             |

- 虚拟代理区括号内的名称即为其下辖的真实代理区（如"浙江代理区（杭州、宁波、温台、金华）"表示杭州代理区、宁波代理区、温台代理区、金华代理区归属浙江虚拟代理区管理）。
- 没有括号的代理区本身就是独立的真实代理区（如上海代理区、福建代理区）。

## Review 审查原则（强制执行）

除了后端研发可执行性之外，Review 阶段还必须从以下维度审查需求：

**原则1：业务合理性优先于技术实现**

不能只看"这个功能能不能做"，更要看"这个功能该不该做"：
- 需求背后的业务逻辑是否合理？
- 是否存在制度风险或合规风险？
- 功能上线后可能带来什么负面影响？

**原则2：考核数据类需求必须严格审查**

当需求涉及考核指标（KPI）相关的报表或数据时，如果产品提出允许业务方对考核数据做"失效"、"作废"、"剔除"、"修改"等操作，必须在 review.md 中**单独列出业务合理性质疑章节**，要求产品确认：
- 具体的使用场景是什么？为什么需要这个功能？
- 哪些角色可以执行该操作？是否存在被考核方自己修改考核数据的利益冲突？
- 是否需要审批流程？（如申请→审核→确认）
- 操作是否可追溯？（操作日志、操作人、操作时间）
- 考核数据的完整性和公正性如何保证？

**原则3：用户使用便捷度**

站在用户实际使用的角度审查：
- 操作流程是否合理？是否有多余步骤可以简化？
- 异常场景下用户的操作路径是否顺畅？
- 不同角色的操作体验是否都考虑到了？

#### Review 必须包含的"第0节：需求介绍"（强制执行）

在 review.md 的正式评审内容（第1~7节）之前，**必须先写第0节"需求介绍"**，用通俗易懂的语言帮助读者快速理解这个需求的全貌。

**目的**：让任何拿到这份 review.md 的人（评审会参与者、未来的自己、接手的同事），不需要去读原始需求文档，就能在3分钟内理解这个需求的核心。

**必须回答的四个问题**：

1. **这个需求是做什么的？**
   - 用通俗语言概括需求核心内容，不要复制粘贴需求文档原文
   - 涉及的关键概念/术语要翻译成大白话（假设读者是刚入职的新人）
   - 如果需求包含多个部分，说清楚各部分之间的关系

2. **为什么需要做这个？**
   - 还原需求背后的真实业务动机（不是照抄"需求背景"，而是解释清楚痛点）
   - 不做会怎样？做了能解决什么问题？
   - 如果是优化类需求，当前方案的痛点是什么？

3. **谁在用这个功能？**
   - 明确目标用户的具体角色（不只是"总部账号"，而是"客服运营团队"、"网点操作员"等）
   - 不同角色使用目的有什么差异？
   - 权限范围是什么？

4. **在极兔的实际业务中，这个功能怎么用？**
   - 结合极兔实际业务场景举例（快递揽收/派送/转运、客服接待/工单处理、网点管理等）
   - 描述典型使用场景：谁在什么时候、什么情况下会用到
   - 如果涉及业务链路/数据流向，用简单的流程图或文字说清楚

**写作要求**：
- 语言风格：像给同事讲需求一样，口语化、不要官话套话
- 不要凭空编造业务细节——如果需求文档没写、自己也不确定的，就标注"（待确认）"
- 涉及数据类需求时，如果手头有实际数据（如日报、现有报表截图），用真实数据举例比编数据更有说服力

## 使用方式

### 命令行方式

```bash
# Windows
.claude/skills/batch-prd-preview/scripts/batch_preview.bat --input-dir ./prds --output-dir ./output

# 或直接使用 Python
python .claude/skills/batch-prd-preview/scripts/batch_preview.py --input-dir ./prds --output-dir ./output
```

### 参数说明

| 参数 | 说明 | 默认值 |
|---|---|---|
| `--input-dir` | 输入目录路径（包含 Word 文件） | 当前目录 |
| `--output-dir` | 输出目录路径 | `./prd-preview-output` |
| `--file-pattern` | 文件匹配模式 | `*.docx` |

### 示例

```bash
# 处理指定目录下的所有 docx 文件
batch_preview.bat --input-dir D:/prds --output-dir D:/preview

# 处理 doc 文件
batch_preview.bat --input-dir ./docs --file-pattern "*.doc"

# 使用当前目录，输出到默认位置
batch_preview.bat
```

## 需求评审(Review)

从开发可执行性角度审查需求文档质量，为每个文档生成独立的评审报告。

### 评审维度

| 维度 | 审查要点 |
|------|---------|
| 需求清晰度 | 描述是否明确、无歧义 |
| 业务规则 | 边界条件、异常场景是否交代清楚 |
| 数据需求 | 数据结构、数据来源、数据量级是否明确 |
| 接口契约 | 入参、出参、错误码是否定义 |
| 性能要求 | QPS、响应时间、数据量是否明确 |
| 安全要求 | 权限、脱敏、风控是否考虑 |

## Claude 执行流程

当用户触发此 skill 时，按以下步骤执行：

### Step 1: 确认输入目录

1. 询问用户输入目录路径（或使用用户指定的路径）
2. 扫描目录中的 Word 文件
3. 输出待处理文件列表，等待用户确认

### Step 2: 执行批量转换

1. 调用 `batch_preview.bat` 执行批量转换
2. 实时显示转换进度
3. 为每个文件生成 `preview.md`

### Step 3: 执行需求评审(Review)

1. 逐一读取转换后的 `preview.md`
2. 按6个评审维度进行分析：
   - 需求清晰度
   - 业务规则
   - 数据需求
   - 接口契约
   - 性能要求
   - 安全要求
3. 为每个文档生成 `review.md`
4. 生成汇总评审报告 `review-summary.md`

### Step 4: 输出评审结果

1. 展示 `review-summary.md` 中的评审表格
2. 列出共性问题
3. 给出优先级建议
4. **写完后停下来，等用户审阅并确认**

## 批注迭代

用户在 `review.md` 中通过批注标记反馈：

| 标记 | 用途 | 示例 |
|------|------|------|
| `> **[批注]**` | 纠正事实、补充信息 | `> **[批注]** 这里少了代理区账号的越权场景` |
| `> **[否决]**` | 否定某个评审结论 | `> **[否决]** 性能要求其实已经明确了` |
| `> **[补充]**` | 添加约束条件 | `> **[补充]** 还需要一个前置条件：网点必须是已激活状态` |
| `> **[疑问]**` | 提出问题 | `> **[疑问]** 这里的并发量预估是多少？` |
| `> **[确认]**` | 表示通过审阅 | `> **[确认]** review没问题，可以进入下一阶段` |

**处理规则**：
- 读到 `[批注]`/`[否决]`/`[补充]`：修改对应评审内容，更新文档
- 读到 `[疑问]`：在文档中回答或直接回复用户
- 读到 `[确认]`：评审阶段结束

---

## 下一步建议

评审完成后，用户可选择：

1. **继续进入 `prd2trd` 流程**：对通过评审的文档执行需求深挖、TRD规划、生成TRD
2. **补充需求信息**：针对评审中发现的问题补充需求文档
3. **重新评审**：需求补充后重新执行评审

## 注意事项

1. **文件命名**：输出目录按原文件名创建子目录
2. **图片处理**：图片统一放在每个文件的 `assets` 子目录
3. **格式兼容**：支持 Pandoc 和 Mammoth 两种转换引擎
4. **错误处理**：单个文件失败不影响其他文件处理
5. **评审独立性**：每个文档独立评审，不互相影响
6. **停等确认**：评审完成后必须停下来等待用户确认，不自动进入下一阶段
7. **与 prd2trd 联动**：评审通过的文档可无缝衔接 `prd2trd` 流程
