# Batch PRD Preview 操作指引

## 一、功能概述

`batch-prd-preview` 是一个批量对需求Word文档进行预审的技能，用于：
- 批量转换 Word 文件（.docx/.doc）为 Markdown 格式
- 从开发可执行性角度评审需求文档质量
- 生成汇总报告，便于快速了解需求全貌

与 `prd2trd` 不同，本技能**只做格式转换和需求评审**，不进入需求深挖、TRD规划、生成TRD等后续流程。

---

## 二、使用方式

### 方式一：直接触发

在 Claude Code 中输入：

```
/batch-prd-preview <目录路径>
```

或：

```
preview <目录路径>
```

示例：
```
preview  prd_docs_0409
```

### 方式二：Skill 工具调用

```
/skill batch-prd-preview prd_docs_0409
```

## 三、操作流程

### Step 1: 扫描文件

系统自动扫描指定目录下的 Word 文件（.docx/.doc），输出待处理文件列表。

```
**扫描到以下 13 个 Word 文档：**

| 序号 | 文件名 |
|------|--------|
| 1 | 【项目-工单质效提升】客服工单工具.docx |
| 2 | 仲裁-责任方主动协商prd-.docx |
| ... | ... |
```

### Step 2: 批量转换

使用 Pandoc 将 Word 文档转换为 Markdown 格式：
- 每个文件单独创建输出目录
- 图片提取到 `assets` 子目录
- 生成 `prd.md` 文件

### Step 3: 需求评审

从 6 个评审维度分析每个文档：
1. **需求清晰度** - 描述是否明确、无歧义
2. **业务规则** - 边界条件、异常场景是否交代清楚
3. **数据需求** - 数据结构、数据来源、数据量级是否明确
4. **接口契约** - 入参、出参、错误码是否定义
5. **性能要求** - QPS、响应时间、数据量是否明确
6. **安全要求** - 权限、脱敏、风控是否考虑

为每个文档生成 `review.md` 评审报告。

### Step 4: 生成汇总报告

生成两个汇总文件：
- `preview-summary.md` - 文件处理概览
- `review-summary.md` - 评审结果汇总

### Step 5: 等待用户确认

**重要**：评审完成后会停下来等待用户确认，不会自动进入下一阶段。

## 四、输出结构

```
prd-preview-output/
├── preview-summary.md          # 预览汇总报告
├── review-summary.md           # 需求评审汇总报告（核心）
├── 文件A/
│   ├── assets/                 # 图片目录
│   ├── prd.md                  # 转换后的 Markdown
│   └── review.md               # 需求评审报告
├── 文件B/
│   ├── assets/
│   ├── prd.md
│   └── review.md
└── ...
```

## 五、评审报告内容

### 5.1 review.md 结构

每个文档的评审报告包含：

```markdown
# 需求评审：[需求名称]

## 0. 需求介绍（必读）

### 这个需求是做什么的？
[通俗语言解释核心内容和关键术语]

### 为什么需要做这个？
[还原业务动机和痛点]

### 谁在用这个功能？
[目标用户角色、权限范围]

### 在极兔实际业务中，这个功能怎么用？
[典型使用场景举例]

---

## 评审维度

### 1. 需求清晰度
[状态：✅/⚠️/❌ + 具体问题]

### 2. 业务规则
[状态：✅/⚠️/❌ + 具体问题]

### 3. 数据需求
[状态：✅/⚠️/❌ + 具体问题]

### 4. 接口契约
[状态：✅/⚠️/❌ + 具体问题]

### 5. 性能要求
[状态：✅/⚠️/❌ + 具体问题]

### 6. 安全要求
[状态：✅/⚠️/❌ + 具体问题]

---

## 评审结论
[通过/需补充/需重写]

## 下一步建议
[P0必须补充项、P1建议补充项]
```

### 5.2 review-summary.md 结构

汇总报告包含：
- **评审概览表格** - 所有文档的6维度评分
- **共性问题分析** - 发现的普遍问题
- **优先级建议** - 按优先级分类的建议
- **资损风险专项** - 涉及资金的需求提醒

---

## 六、评审维度说明

| 维度 | 状态符号 | 审查要点 |
|------|---------|---------|
| **需求清晰度** | ✅ ⚠️ ❌ | 功能范围明确、术语解释清晰、无歧义表述 |
| **业务规则** | ✅ ⚠️ ❌ | 边界条件、异常场景、并发场景、状态流转 |
| **数据需求** | ✅ ⚠️ ❌ | 表结构、字段类型、数据量级、历史兼容 |
| **接口契约** | ✅ ⚠️ ❌ | 入参、出参、错误码、幂等设计 |
| **性能要求** | ✅ ⚠️ ❌ | QPS、响应时间、并发量、压测方案 |
| **安全要求** | ✅ ⚠️ ❌ | 权限校验、越权风险、脱敏、防刷 |

## 七、批注迭代机制

评审完成后，用户可在 `review.md` 中通过批注标记反馈：

| 标记 | 用途 | 示例 |
|------|------|------|
| `> **[批注]**` | 纠正事实、补充信息 | `> **[批注]** 这里少了代理区账号的越权场景` |
| `> **[否决]**` | 否定某个评审结论 | `> **[否决]** 性能要求其实已经明确了` |
| `> **[补充]**` | 添加约束条件 | `> **[补充]** 还需要一个前置条件：网点必须是已激活状态` |
| `> **[疑问]**` | 提出问题 | `> **[疑问]** 这里的并发量预估是多少？` |
| `> **[确认]**` | 表示通过审阅 | `> **[确认]** review没问题，可以进入下一阶段` |

**处理规则**：
- `[批注]`/`[否决]`/`[补充]` → 修改对应评审内容，更新文档
- `[疑问]` → 在文档中回答或直接回复用户
- `[确认]` → 评审阶段结束

---

## 八、下一步选择

评审完成后，用户可选择：

### 选项 1：进入 prd2trd 流程

对通过评审的文档执行：
```
prd2trd <文档目录>
```

进入需求深挖 → TRD规划 → 生成TRD的完整流程。

### 选项 2：补充需求信息

针对评审中发现的问题，补充需求文档内容后重新评审：
```
preview <目录路径>
```

### 选项 3：重新评审

需求补充后重新执行评审流程。

## 九、注意事项

1. **文件命名** - 输出目录按原文件名创建，中文文件名会保留
2. **图片处理** - 图片统一放在每个文件的 `assets` 子目录
3. **格式兼容** - 支持 .docx 和 .doc 两种格式
4. **错误处理** - 单个文件失败不影响其他文件处理
5. **评审独立性** - 每个文档独立评审，不互相影响
6. **停等确认** - 评审完成后必须停下来等待用户确认
7. **环境依赖** - 需要安装 Pandoc（推荐 3.x 版本）

## 十、常见问题

### Q1: 如何查看评审结果？

进入输出目录查看：
```
prd-preview-output/review-summary.md  # 汇总报告
prd-preview-output/文件名/review.md   # 详细评审
```

### Q2: 评审结论"需补充"怎么办？

查看对应 `review.md` 中的：
- **必须补充项（P0）** - 开发前必须明确的内容
- **建议补充项（P1）** - 提升需求质量的内容

### Q3: 如何进入 TRD 规划？

确认评审通过后：
```
prd2trd prd-preview-output/文件名
```

### Q4: 文件转换失败怎么办？

检查：
1. Word 文件是否损坏
2. Pandoc 是否正确安装
3. 文件路径是否正确
