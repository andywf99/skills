# PRD 转 TRD Skill

将 PRD（产品需求文档）转换为 TRD（概要设计文档）的技能，遵循极兔速递的概要设计规范。

---

## 文件夹结构约定

每个需求对应一个文件夹，所有产出物集中存放：

```
#12xxxxx 【业务需求】XXX/
├── assets/                          ← 需求文档中的图片
├── 需求文档.docx                    ← 原始需求文档
├── 需求文档.md                      ← 转换后的markdown（Claude读这个）
├── review.md                        ← 第零步：需求评审
├── research.md                      ← 第一步：需求深挖
├── plan.md                          ← 第二步：TRD规划
└── 概要设计.md                      ← 最终交付物
```

---

## 工作流程

```
格式转换(Format) → 需求评审(Review) → 需求深挖(Research) → TRD规划(Planning) → 批注迭代 → 生成TRD
```

### Step 0: 文件格式转换

**触发条件**：用户提供 PRD 文档（.docx/.doc 格式）

**处理流程**：
1. 用户将 PRD 文档放置到当前项目的根目录中
2. 根据 PRD文档名称生成文件夹“#12xxxxx 【业务需求】XXX” 
3. 移动原需求文档文件到新文件夹中
4. 使用 `word2md.bat` 工具转换为 Markdown 格式
4. 转换命令：
   ```bash
   .claude/skills/prd2trd/scripts/word2md.bat "#12xxxxx 【业务需求】XXX/需求文档.docx" "#12xxxxx 【业务需求】XXX/需求文档.md"
   ```
5. Claude 读取转换后的 Markdown 文件

**注意事项**：
- 仅执行脚本即可，不需要调整 [word2md_cli.py](scripts/word2md_cli.py) 中的代码逻辑
- 转换后检查图片是否正确提取到 `assets` 目录
- 检查表格、列表等格式是否正确
- 如有格式问题，手动调整或提醒用户

---

### Step 1: 需求评审(Review)

**目的**：从开发可执行性角度审查需求文档质量

**输出**：`review.md`

**审查维度**：

| 维度 | 审查要点 |
|------|---------|
| 需求清晰度 | 描述是否明确、无歧义 |
| 业务规则 | 边界条件、异常场景是否交代清楚 |
| 数据需求 | 数据结构、数据来源、数据量级是否明确 |
| 接口契约 | 入参、出参、错误码是否定义 |
| 性能要求 | QPS、响应时间、数据量是否明确 |
| 安全要求 | 权限、脱敏、风控是否考虑 |

**写完后停下来，等用户审阅并确认。**

---

### Step 2: 需求深挖(Research)

**前提**：用户确认 `review.md` 通过

**目的**：深入分析需求，补充技术细节

**输出**：`research.md`

**分析内容**：
1. 功能点拆解
2. 业务规则梳理
3. 数据模型分析
4. 接口依赖梳理
5. 技术风险识别

**写完后停下来，等用户审阅并确认。**

---

### Step 3: TRD规划(Planning)

**前提**：用户确认 `research.md` 通过

**目的**：按照概要设计规范规划 TRD 结构

**输出**：`plan.md`

**规划内容**：
- 各章节的内容大纲
- 需要补充的技术细节
- 待确认的技术决策

**写完后停下来，等用户审阅并确认。**

---

### Step 4: 批注迭代

用户在 `review.md`、`research.md` 或 `plan.md` 中通过批注标记反馈：

| 标记 | 用途 | 示例 |
|------|------|------|
| `> **[批注]**` | 纠正事实、补充信息 | `> **[批注]** 这里少了代理区账号的越权场景` |
| `> **[否决]**` | 否定某个方案 | `> **[否决]** 这个方案不行，性能有问题` |
| `> **[补充]**` | 添加约束条件 | `> **[补充]** 还需要一个前置条件：网点必须是已激活状态` |
| `> **[疑问]**` | 提出问题 | `> **[疑问]** 这里的并发量预估是多少？` |
| `> **[确认]**` | 表示通过审阅 | `> **[确认]** research没问题，可以出plan了` |

**处理规则**：
- 读到 `[批注]`/`[否决]`/`[补充]`：修改对应内容，更新文档
- 读到 `[疑问]`：在文档中回答或直接回复用户
- 读到 `[确认]`：进入下一阶段
- **没有读到 `[确认]`，绝不进入下一阶段**

---

### Step 5: 生成TRD

**前提**：用户确认 `plan.md` 通过

**输出**：`概要设计.md`

**遵循极兔概要设计规范**，见下方模板。

## 极兔速递背景知识

### 一、组织架构

#### 1.1 组织层级结构

```
总部（最高层级）
  ├── 代理区
  │   ├── 虚拟代理区（下辖多个真实代理区）
  │   │   ├── 浙江代理区（杭州、宁波、温台、金华）
  │   │   ├── 广东代理区（广州、深圳、东莞、粤西、粤东）
  │   │   ├── 京津冀代理区（河北、天津、北京）
  │   │   ├── 甘青宁代理区（甘肃、青海、宁夏）
  │   │   ├── 江苏代理区（苏北、苏南）
  │   │   ├── 渝贵代理区（重庆、贵州）
  │   │   └── 东北代理区（吉林、黑龙江、辽宁）
  │   │
  │   └── 真实代理区（本身即独立代理区，无上级虚拟代理区）
  │       ├── 上海代理区、福建代理区、广西代理区、云南代理区
  │       ├── 新疆代理区、安徽代理区、山东代理区、河南代理区
  │       ├── 山西代理区、湖南代理区、川藏代理区、江西代理区
  │       ├── 陕西代理区、海南代理区、湖北代理区、内蒙古代理区
  │       └── 西藏代理区
  │
  ├── 转运中心（中心层级）
  │   └── 例如：上海浦西转运中心
  │
  ├── 加盟商（中心层级）
  │   └── 例如：上海加盟商三百七十七
  │
  └── 网点（最底层级）
      └── 例如：上海北青公路网点
```
#### 1.2 层级说明

| 层级           | 说明                                         | 示例                                             |
| -------------- | -------------------------------------------- | ------------------------------------------------ |
| **总部**       | 最高管理层级，负责全国业务统筹               | 极兔速递总部                                     |
| **虚拟代理区** | 多个真实代理区合并管理的大区                 | 浙江代理区（下辖杭州、宁波、温台、金华代理区）   |
| **真实代理区** | 实际运营的代理区，可独立或归属虚拟代理区     | 杭州代理区（属于浙江虚拟代理区）、上海代理区（独立） |
| **转运中心**   | 货物中转处理中心，属于中心层级               | 上海浦西转运中心                                 |
| **加盟商**     | 加盟合作伙伴，属于中心层级                   | 上海加盟商三百七十七                             |
| **网点**       | 最基层的服务网点                             | 上海北青公路网点                                 |

**注意**：
- 转运中心和加盟商都属于"中心层级"，在系统中具有相同的层级权限。
- 虚拟代理区括号内的名称即为其下辖的真实代理区（如"浙江代理区（杭州、宁波、温台、金华）"表示杭州代理区、宁波代理区、温台代理区、金华代理区归属浙江虚拟代理区管理）。
- 没有括号的代理区本身就是独立的真实代理区（如上海代理区、福建代理区）。

### 三、业务术语和概念

#### 3.1 核心业务模块

根据工作文档，主要涉及以下业务模块：

1. **无头件管理**
   - 无头件登记、认领、审核
   - 无头件内物信息查询
   - 残单检索
   - 作废件处理

2. **问题件管理**
   - 问题件登记、回复、查询
   - 问题件转作废、转留仓
   - 问题件监控

3. **工单系统**
   - **仲裁工单**：处理快件纠纷和判责
   - **质量工单**：处理质量问题
   - **监控工单**：处理监控发现的问题
   - **平台工单**：处理平台相关工单

4. **完结件管理**
   - 完结件记录生成
   - 完结网点取值逻辑
   - 不同来源数据进入完结件

5. **作废件管理**
   - 无头件自动转作废
   - 作废件审核流程
   - 作废件处理

#### 3.2 常见业务术语

| 术语             | 说明                                                   |
| ---------------- | ------------------------------------------------------ |
| **无头件**       | 有实物但没有运单（面单丢失/损坏）的包裹                |
| **残单检索**     | 通过描述物品信息（收寄件人、地址等）查找运单号         |
| **内物信息查询** | 通过运单号查询包裹内物品信息（商品名称、照片、价值等） |
| **作废件**       | 无法继续派送需要作废处理的快件                         |
| **完结件**       | 记录快件异常完结状态的记录                             |
| **问题件**       | 在运输过程中出现问题的快件                             |
| **仲裁工单**     | 处理快件纠纷和责任判定的工单                           |
| **质量工单**     | 处理质量问题的工单                                     |
| **监控工单**     | 通过监控发现问题的工单                                 |
| **平台工单**     | 来自电商平台的工单                                     |
| **代理区转责**   | 代理区重新划分责任网点的操作                           |
| **认领**         | 网点认领无头件的过程                                   |
| **审核**         | 对认领进行审核确认的过程                               |

#### 3.3 数据来源说明

在无头件内物查询中，数据来源包括：

- **菜鸟接口**：外部平台接口
- **平台工单**：极兔内部平台工单系统
- **仲裁工单**：极兔内部仲裁工单系统
- **客户物品信息库**：人工导入的客户物品信息

## Review 审查原则（强制执行）

除了后端研发可执行性之外，Review 阶段还必须从以下维度审查需求：

**原则1：业务合理性优先于技术实现**

不能只看"这个功能能不能做"，更要看"这个功能该不该做"：
- 需求背后的业务逻辑是否合理？
- 是否存在制度风险或合规风险？
- 功能上线后可能带来什么负面影响？

**原则2：考核数据类需求必须严格审查**

当需求涉及考核指标（KPI）相关的报表或数据时，如果产品提出允许业务方对考核数据做"失效"、"作废"、"剔除"、"修改"等操作，必须在 review.md 中**单独列出业务合理性质疑章节**，要求产品确认：
- 具体的使用场景是什么？为什么需要这个功能？
- 哪些角色可以执行该操作？是否存在被考核方自己修改考核数据的利益冲突？
- 是否需要审批流程？（如申请→审核→确认）
- 操作是否可追溯？（操作日志、操作人、操作时间）
- 考核数据的完整性和公正性如何保证？

**原则3：用户使用便捷度**

站在用户实际使用的角度审查：
- 操作流程是否合理？是否有多余步骤可以简化？
- 异常场景下用户的操作路径是否顺畅？
- 不同角色的操作体验是否都考虑到了？

#### Review 必须包含的"第0节：需求介绍"（强制执行）

在 review.md 的正式评审内容（第1~7节）之前，**必须先写第0节"需求介绍"**，用通俗易懂的语言帮助读者快速理解这个需求的全貌。

**目的**：让任何拿到这份 review.md 的人（评审会参与者、未来的自己、接手的同事），不需要去读原始需求文档，就能在3分钟内理解这个需求的核心。

**必须回答的四个问题**：

1. **这个需求是做什么的？**
   - 用通俗语言概括需求核心内容，不要复制粘贴需求文档原文
   - 涉及的关键概念/术语要翻译成大白话（假设读者是刚入职的新人）
   - 如果需求包含多个部分，说清楚各部分之间的关系

2. **为什么需要做这个？**
   - 还原需求背后的真实业务动机（不是照抄"需求背景"，而是解释清楚痛点）
   - 不做会怎样？做了能解决什么问题？
   - 如果是优化类需求，当前方案的痛点是什么？

3. **谁在用这个功能？**
   - 明确目标用户的具体角色（不只是"总部账号"，而是"客服运营团队"、"网点操作员"等）
   - 不同角色使用目的有什么差异？
   - 权限范围是什么？

4. **在极兔的实际业务中，这个功能怎么用？**
   - 结合极兔实际业务场景举例（快递揽收/派送/转运、客服接待/工单处理、网点管理等）
   - 描述典型使用场景：谁在什么时候、什么情况下会用到
   - 如果涉及业务链路/数据流向，用简单的流程图或文字说清楚

**写作要求**：
- 语言风格：像给同事讲需求一样，口语化、不要官话套话
- 不要凭空编造业务细节——如果需求文档没写、自己也不确定的，就标注"（待确认）"
- 涉及数据类需求时，如果手头有实际数据（如日报、现有报表截图），用真实数据举例比编数据更有说服力

- 涉及数据类需求时，如果手头有实际数据（如日报、现有报表截图），用真实数据举例比编数据更有说服力

### 3. 数据结构

**要求**：
- 数据库表结构：定义 E-R 模型，输出 DDL 语句（合理设置表字段长度上限、索引）
- 数据幂等设计：所有表都要保证有唯一索引，防止并发导致重复数据入库
- 特别注意：工单类、订单类、配置类、流水类、短信类业务必须有唯一索引
- 无法增加唯一索引的业务需求，必须提供充足的理由并审核通过

**输出内容**：
- 表结构设计
- DDL语句
- 索引设计
- 幂等方案

### 4. 数据回刷

参考：[服务质量-刷数据影响范围评估规范](https://jtexpress.feishu.cn/wiki/QqpQwBgO7iLTVVkiO0PcO8H9n1d)

**输出内容**：
- 回刷场景说明
- 影响范围评估
- 回刷脚本
- 回滚方案

### 5. 接口设计

**要求**：
- 接口文档：接口url、入参、出参
- 并发设计：新增、变更类必须要有分布式乐观锁/悲观锁
- 事务设计：多表新增或更新场景必须增加事务
- 日志设计：新增、变更类必须要有详细日志留痕

**输出内容**：
- 接口列表
- 接口详细设计（URL、方法、入参、出参、错误码）
- 并发控制方案
- 事务边界
- 日志埋点

### 6. 性能设计

**要求**：
- 接口压测：所有接口必须在需求评审阶段确认性能指标
- 外域接口：提供给第三方、外域的接口必须经过平台压测，上线后使用 Sentinel 限流
- 性能考虑：确认是否会出现慢查询？是否存在 OOM？
- 业务并发量大的读写场景，必须汇报给旗主进行方案及压测结果确认
- 数据限制：导入、导出、批处理接口是否有数量限制？

**输出内容**：
- 性能指标要求
- 压测方案
- 慢查询分析
- OOM 风险评估
- 分表/ES/ADB 等中间件使用方案

### 7. 容错设计

**要求**：
- 异步异常预警：异步/JOB 中必须对代码块进行异常捕获，并在异常时飞书告警
- 数据增量预警：数据量突增或超出阈值时需告警
- 避免批处理阻塞：批量处理时，单条数据出现已知异常需跳过并告警，不可阻塞后续处理

**输出内容**：
- 异常处理方案
- 告警机制
- 批处理容错方案

### 8. 灰度设计

参考：[灰度发布](https://jtexpress.feishu.cn/wiki/YG8SwLwnmiqotgkE0O1cAyPgnNg)

**输出内容**：
- 灰度发布方案
- 回滚方案
- 灰度开关配置

### 9. 安全设计

**要求**：
- 权限校验：需保证无水平和垂直越权
- 数据脱敏：包含但不限于手机号、身份证号、银行卡号、收寄件地址、邮箱

**输出内容**：
- 权限设计
- 脱敏方案
- 安全风险评估
