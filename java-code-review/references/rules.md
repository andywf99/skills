# 审查清单

审查生成或改动的代码时，将本文件作为详细规则参考。

## 灰度开关

- **必须**：对于历史代码修改（包含在原有方法中新增代码逻辑），需全量覆盖灰度开关，并确保可通过灰度开关切换新旧版本代码。
  - 方法级灰度开关使用规范：`@Gray("xxxx.xxxx")`
  - 代码块级灰度开关使用规范：`GrayUtils.isGray`
- **必须**：对于中台项目中新的 Controller 接口，需新增灰度开关。

## Elasticsearch

- 索引命名：禁止使用大写字母和特殊字符。
- 文档写入：禁止单条循环写入，必须使用 Bulk API；单批文档数不超过 1000 条。
- 文档更新：优先使用 doc 部分更新，避免先查询再全量覆盖。
- 禁止通配符开头：禁止使用 `*xxx` 或 `?xxx` 开头的通配符查询，会导致全索引扫描，性能极差。
- 禁止深分页：禁止使用 `from + size > 10000` 的深分页；大数据量分页必须使用 `search_after` 或 `scroll`。
- 限制返回字段：使用 `_source` 过滤只返回需要的字段，禁止返回全量文档。
- 限制返回条数：查询必须设置 `size` 上限，默认不超过 10000，避免一次性返回过多数据。
- 禁止高基数聚合：禁止对高基数字段（如订单号、运单号、工单号、用户ID等）做 Terms 聚合，会导致内存溢出。
- 聚合分桶限制：Terms 聚合必须设置 `size` 限制分桶数量，默认不超过 1000。
- 脚本查询：禁止在查询中使用脚本（script），性能极差且存在安全风险。
- 索引模式查询：禁止使用 `index_*` 等模糊模式跨过多索引查询，必须明确指定索引。
- 游标查询：大量数据导出使用 Scroll API 或 PIT + Search After，查询完成后必须清除游标。
- Filter 优先：精确过滤场景优先使用 filter 上下文，可利用缓存且不计算得分。
- **禁止 scroll 不规范用法**：
  - 使用 elasticsearch 查询时，为每个 scroll 显式设置合理的 scrollTimeout（例如1-2分钟），避免永久驻留。
  - 在 finally 块中无条件需显式调用 `clearScroll()` 方法清理上下文以释放缓存。

## RocketMQ

- 循环发送：不推荐在循环中逐条发送消息，可使用批量发送；单批数量不超过50条；批量操作需考虑部分成功部分失败场景。
- 幂等性：消费逻辑必须实现幂等性，推荐方案：数据库唯一索引、业务状态机判断。
- 禁止吞异常：禁止 catch 异常后不做任何处理直接返回成功。
- 禁止阻塞操作：消费逻辑中禁止使用 `Thread.sleep()`、无限等待锁、长耗时 IO 等阻塞操作。
- Topic 命名：Topic 名称总字符数不得超过32个字符。
- 序列化格式：禁止使用 Java 原生序列化。

## 数据库使用

- SQL 安全：禁止字符串拼接 SQL、`${}`、动态拼接 order by、`Wrapper.last(...)` 直接带入外部参数；必须使用 `#{}`、预编译参数或白名单映射。
- 单条查询：使用 `getOne`、`selectOne`、`one`、`findFirst`、`limit 1` 时，查询条件必须保证结果天然唯一；禁止用"取第一条"掩盖脏数据。
- 排序稳定性：单条查询使用 `limit 1`、分页查询使用 `limit/offset` 或分页插件时，必须显式指定稳定 `order by`；禁止无排序返回不确定结果。
- 字段收敛：禁止 `select *`；查询必须按需选择字段，尤其是分页、大表、联表、存在性判断场景。
- 存在性判断：仅判断数据是否存在时，禁止查询完整列表或完整实体；优先使用 `exists`、`count`、`limit 1` 或最小字段查询。
- 结果量控制：无分页列表查询、导出查询、批处理扫描、定时任务扫描时，必须限制结果量；禁止一次性全量拉取大批数据。
- 深分页控制：禁止默认使用大 offset、大页码深分页；数据量大时必须评估游标分页、主键翻页或覆盖索引方案。
- 循环查库：禁止在循环体内逐条调用 mapper、repository、service 查询；必须改为批量查询后在内存中组装。
- IN 使用：in 查询必须做空集合保护；禁止直接传空集合；大集合 in 必须分批处理，避免 SQL 过长和执行性能下降。
- 索引命中：禁止在索引列上做函数、表达式、计算或类型不一致比较，如 `date(column)`、`substring(column)`、`cast(column)`、字符串比数字；必须保证查询可走索引。
- 时间范围：按天、小时、账期查询时，禁止使用 `between` 配合 `23:59:59` 作为结束条件；必须优先使用左闭右开区间：`>= start and < nextStart`。
- 动态排序：排序字段、排序方向来自前端、入参、DTO 时，禁止直接拼接到 SQL；必须做字段白名单和排序方向枚举校验。
- 联表复杂度：多表 join、复杂 XML SQL、自定义联表分页时，必须审查过滤条件、字段范围和结果集规模；禁止弱条件下大宽表联查。
- 统计准确性：`count(*)`、`count(1)` 与 join、group by、distinct 组合使用时，必须确认统计口径与业务语义一致；禁止默认认为总数一定准确。
- 内存过滤下推：禁止先全量查出列表，再用 `stream().filter/sort/skip/limit` 做主要筛选、排序、分页；可下推到 SQL 的逻辑必须优先在数据库层完成。

## Redis 使用规范

### 基础使用规范

1. Key 必须统一使用业务名或库名作为前缀，并用冒号分隔，避免不同系统间出现命名冲突。
2. Key 命名必须清晰表达业务含义，禁止使用无意义缩写，方便开发、测试和运维理解。
3. 所有业务缓存原则上都必须设置过期时间，防止脏数据长期存在并持续占用 Redis 内存。
4. 禁止在 Redis 中存储大 Key 或大 Value，避免网络传输、序列化和访问性能明显下降。
5. 线上环境禁止使用 `KEYS` 等全量扫描命令，防止阻塞 Redis，影响其他业务正常访问。

### 项目层面使用规范

6. 使用 Redis 分布式锁时必须设置过期时间，防止服务异常后锁无法释放而导致死锁问题。
7. 释放分布式锁时必须校验唯一标识，禁止直接删除锁，避免误删其他线程持有的锁。
8. 所有加锁代码都必须使用 `try-finally` 释放锁，确保业务异常时也不会遗漏解锁操作。
9. 分布式锁内部禁止执行远程调用、慢查询等耗时操作，避免锁超时失效引发并发问题。
10. 并发场景下禁止使用 `get-then-set` 更新数据，应优先使用 `INCR`、`DECR` 或 Lua 保证原子性。
11. 在 Spring 事务中操作 Redis 时必须注意一致性，避免数据库回滚后缓存数据已被提前修改。

## 资源关闭 (Try-with-resources)

- 涉及 IO 流（FileInputStream 等）必须使用 `try-with-resources` 语法。
- **错误**：在 `finally` 中手动 close 或未关闭。

## 多线程安全

- 在 `executorService.execute(() -> {...})` 或线程池任务中，内部必须包裹 `try-catch` 并打印错误日志，防止异常吞没。RocketMQ 消息监听的误判排除。

## 空指针防御 (NPE)

- 禁止直接调用可能为 null 对象的方法。
- 两个数据比较时，必须使用 `Objects.equals(A, B)`，严禁直接使用 `A.equals(B)`（除非 A 确认为常量）。

## 事务管理

- 涉及多表/多行增删改操作，必须添加 `@Transactional` 注解或显式事务的代码块。

## 日志规范

- **禁止**：`System.out.println` 或 `e.printStackTrace()`。
- **必须**：使用 `@Slf4j` (log.info/error)。

## 禁止循环查库

- 严禁在循环(for/foreach/while)中调用数据库查询或网络 RPC（有明确分批处理的除外，例如：`for (List<String> waybillNos : partitions) {}`）

## 禁止多次循环同一个集合

- 严禁在同一个方法中3次及以上循环同一个集合。

## 常规规范

### 1. 批量/导入/导出 - 边界控制

1.1 批量操作（List参数）必须检查大小是否超过上限（建议≤1000），超过时拒绝或分片处理
1.2 文件导入必须检查文件大小，超过阈值（如10MB）时提示或异步处理
1.3 批量删除/更新必须包含WHERE条件，禁止无条件全表操作

### 2. 并发场景 - 悲观/乐观锁

2.1 高并发更新场景必须使用分布式锁
2.2 多个锁获取必须保持全局一致顺序，避免死锁

### 3. 事务管理

3.1 事务中禁止包含RPC调用、HTTP请求、消息发送等可能长时间阻塞的操作
3.2 `@Transactional` 必须指定 `rollbackFor=Exception.class`，避免回滚失败
3.3 事务方法必须是 public，内部调用不会触发事务代理（需注意自调用问题）
3.4 禁止在事务内执行大量非DB操作或循环更新（改为分批提交）

### 4. 空指针与边界条件

4.1 所有方法参数、返回值在使用前必须判空（或使用 `@NotNull`/Optional 明确语义）
4.2 链式调用（`a.b().c()`）必须在每步判空
4.3 除法/取模运算必须检查除数为0
4.4 集合/数组通过索引访问前必须检查 `size>index`

### 5. 异常处理

5.1 禁止捕获 Exception 后不处理（空catch块）或不记录日志
5.2 禁止使用 `e.printStackTrace()`，必须使用日志框架记录
5.3 finally 块中不应抛出异常，避免掩盖原始异常

### 6. 日志规范

6.1 禁止日志中打印密码、Token、身份证号、银行卡号等敏感信息
6.2 必须使用占位符（`{}`）而非字符串拼接
6.3 异常日志必须包含异常堆栈（`log.error("msg", e)`）

### 7. 性能与资源

7.1 循环体内禁止执行RPC调用、数据库查询（改为批量或缓存）
7.2 所有 InputStream/Connection/ResultSet 必须在 finally 或 try-with-resources 中关闭
7.3 大字符串拼接使用 StringBuilder 而非 `+` 号拼接
7.4 避免频繁创建大对象（如 `new HashMap` 在循环内）

### 8. 代码可读性

8.1 方法长度超过80行建议拆分
8.2 参数数量超过5个建议封装为对象
8.3 嵌套层级超过5层建议扁平化或提取方法
8.4 禁止存在明显的重复代码（相同逻辑≥2处）
8.5 复杂逻辑必须有注释说明

## 忽略规则

以下情况在审查时应当忽略，不计入违规项：

1. **注解值过长**：如 `@Value`, `@RabbitListener(queues = "...")`, `@RequestMapping` 等注解内的字符串无论多长，均**完全忽略**。
2. **日志拼接**：使用 `+` 号拼接字符串的日志（如 `log.info("ID: " + id)`）是允许的，**不要**要求改为占位符。
3. **非新增代码**：再次强调，`type` 不为 `"new"` 的所有行，无论任何情况，均忽略。
4. **工具类保护**：如果工具类有校验 null 等情况，忽略但调用处应增加保护的检查。
5. **Lombok 使用**：代码中已经使用 lombok 相关，忽略相关的 getter/setter 等方法缺失问题。
6. **已存在 null 判断**：如果已经存在 `if (Objects.nonNull(变量))` 判断，则不应该再提示 NPE 问题。
7. **消息处理异常**：消息处理方法缺少异常处理，可能导致消息消费失败 或者 方法调用缺少异常处理，可能导致消息消费失败。这个均**完全忽略**，且不扣分。
8. **日期注释**：当出现 类注释包含 `@date` 注释时候以及其他注释时间，这个检查内容忽略检测、不返回审查结果，不扣分。
9. **Controller 层事务**：在 Controller 层不需要、禁止检测 `@Transactional` 注解。
10. **@date 注释**：类注释包含 `@date` 注释，禁止检测此注解，允许存在。也不用返回到核心问题中。
11. **SQL 硬编码数字**：忽略 SQL 脚本中存在硬编码的数字值（如 '0'），建议使用常量或枚举替代，以上禁止检测。
12. **特定字段名称**：对于已对外暴露、涉及接口兼容性的历史字段，允许忽略字段名拼写检查，不强制修改。
13. **注解中的值**：注解中的值不需要校验是否为常量，例如 `@Size(max = 200, message = "单次查询运单号不能超过 200 条")`，禁止检测。
14. **数量字段类型**：数量字段不强制使用 BigDecimal，禁止检测。
15. **通配符导入**：禁止使用通配符导入，应明确列出具体类，禁止检测。
16. **ArrayList 导入**：ArrayList 但未见导入语句，可能导致编译错误，建议改用 List 接口或补充 import java.util.ArrayList; 这个均**完全忽略**，且不扣分。

## 严重级别参考

- **High**：可能导致全表或全索引扫描、数据损坏、死锁、缓存不一致、内存暴涨、敏感信息暴露或明显的生产稳定性问题。评分权重：10。
- **Medium**：可能导致结果不稳定、额外负载、分页错误、边界检查缺失或持续性的性能退化。评分权重：6。
- **Low**：主要是可读性弱、复杂逻辑缺少必要注释，或暂时不直接影响正确性与稳定性的可维护性风险。评分权重：3。
