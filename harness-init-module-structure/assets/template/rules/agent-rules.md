# Agent 工作规则

## 适用范围

这份文档承接 `__MODULE_NAME__` 模块内 Agent 的详细工作规则。`AGENTS.md` 只负责指路和索引，具体规则集中写在这里，并按需加载。

## 工作边界

- 只在 `__MODULE_NAME__` 模块内部工作。
- 优先使用模块内的文档、规范、技能和运行骨架。
- 不假设模块外存在可直接感知的上层目录信息。
- 当前阶段以结构和文档落地为主，除非实际实现了运行时，否则不要暗示模块已经具备可执行的多 Agent 能力。

## 输出约定

- Harness 架构说明和运行说明放在 `.harness/`。
- 跨任务复用的 Agent、工程交付和代码审查规则放在 `rules/`。
- 产品白皮书、技术白皮书等领域长期文档放在 `docs/domain/`。
- 需求改动对系统的影响记忆放在 `docs/memory/memory.md`。
- 需求材料和结构化约束统一放在 `docs/memory/demand/<demand-name>/`。
- 需求归档摘要统一放在 `docs/memory/summary/`。
- 模块私有技能定义放在 `skills/`。
- 面向运行时的骨架和校验脚本放在 `codes/`。
- 会跨任务复用的详细规则优先沉淀到 `rules/`，`AGENTS.md` 只保留入口和索引。

## 工程交付规则入口

- 需要遵循通用工程交付规则时，加载 `rules/engineering-rules.md`。
- `agent-rules.md` 管模块工作边界、产物落位和长期记忆约束；`engineering-rules.md` 管工程实现、兼容性、SQL、日志、安全和发布等通用交付要求。
- 如果两者冲突，以模块边界和更高优先级约束为准。

## 项目代码获取规则

- `__MODULE_NAME__` 模块绑定的项目代码仓信息写在 `AGENTS.md` 的“绑定项目代码”章节。
- 当 Agent 进入模块后，如果执行任务时发现本地缺少对应项目代码，应优先检查 `codes/` 下是否已存在同名目录。
- 如果 `codes/` 下不存在对应代码目录，则使用 `AGENTS.md` 中记录的 Git 地址执行 `git clone`，并将代码拉取到 `codes/` 目录下。
- 代码仓目录名应与仓库名保持一致，便于后续脚本和人工定位。

## 系统影响记忆规则

- 每次需求改动都应评估它对系统的影响，并记录到 `docs/memory/memory.md`。
- 记录的最小粒度是接口，不要只写到系统或模块级别。
- 至少要写清楚接口名称、变更类型、接口功能和影响说明。

### 推荐格式

```text
- 日期：
  需求：
  项目：
  变更类型：新增 / 修改 / 删除
  接口：
  接口功能：
  影响说明：
```

### 示例

- 增加了 `AAA` 接口，接口功能是 `XXX`
- 删除了 `AAA` 接口

## 稳定决策规则

- 保留现有顶层目录，在原位增量扩展。
- 六层 Harness 主要通过 `codes/` 目录表达。
- 模块边界必须清晰、可检查、可解释。

## 已知注意事项

- 空目录在版本控制中容易丢失；当前模块允许通过 `.module-manifest.yaml` 和 `AGENTS.md` 记录空目录职责，必要时再补显式跟踪文件。
- 架构说明和长期记忆不是一回事，不能混写到同一个文件里。
- 详细规则不要继续堆进 `AGENTS.md`，应先判断是否属于 `.harness/`、`rules/` 或 `docs/` 中已有文档的范畴。
