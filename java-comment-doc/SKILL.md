# Java 注释与 JavaDoc 规范技能

按以下流程执行，不要跳步。

## 1. 先判断注释范围

- 默认优先处理 `public` 和 `protected` 方法，再按需处理私有方法。
- 如果用户只要求“补注释”，优先补类注释、方法注释和关键业务行内注释，不要把每一行代码都加上机械说明。
- 如果用户明确要求 `JavaDoc`、接口文档化、参数说明，或者方法本身较复杂，再补完整的 `@param`、`@return`、`@throws` 和参数示例。

## 2. 类、字段、方法的处理原则

### 类注释

- 说明类的业务职责，不要只写“服务类”“工具类”。
- 如果类承担聚合、编排、适配、转换等角色，点明边界。
- Spring 组件、MQ 消费者、定时任务、策略实现类等，要交代触发入口或调用场景。

### 字段注释

- 仅为含义不直观、业务约束强、状态枚举类字段补注释。
- 对显而易见的本地变量、`logger`、框架注入字段不要堆砌注释。

### 方法注释

- 优先解释“做什么”和“为什么这样做”。
- 避免复述代码字面意思，例如“设置名称”“循环列表”这类低价值注释。
- 如果已有注释基本正确，只修正错误、缺失或过时内容，不要无意义重写。

## 3. 什么时候写完整 JavaDoc

满足任一条件时，补完整 JavaDoc：

- 用户明确要求“写 JavaDoc”或“规范接口注释”。
- 方法是对外接口、核心服务、公共组件、抽象方法或模板方法。
- 参数含义不直观，或者需要说明约束、状态、幂等、事务、副作用。

推荐模板：

```java
/**
 * 作用：一句话说明方法的业务目的。
 * 说明：按需补充前置条件、状态约束、幂等要求、事务边界或调用时机。
 *
 * 参数示例（JSON）：
 * <pre>{@code
 * {
 *   "requestNo": "REQ-20260319-001",
 *   "tenantId": 1001
 * }
 * }</pre>
 *
 * @param requestNo 请求唯一号，用于幂等控制
 * @param tenantId 租户标识
 * @return 创建后的结果对象
 * @throws BizException 当参数非法或状态不允许时抛出
 */
```

补充规则：

- 参数示例字段名应与方法参数名一致。
- 参数是对象时，只展示关键业务字段，不必把整个对象结构展开成冗长文档。
- 无参方法只有在用户明确要求完整 JavaDoc 或方法语义复杂时，才补 `参数示例（JSON）`，格式使用空对象 `{}`。

## 4. 方法体内注释怎么写

- 只在关键节点补注释：参数校验、状态流转、权限判断、幂等去重、外部调用、异常兜底、补偿逻辑。
- 注释写在代码块前面，简洁说明该步骤的业务意图。
- 不要出现“调用方法”“处理数据”“执行逻辑”这类空话。

示例：

```java
// 先校验工单当前状态，避免已完结数据被重复派发。
validateStatus(workOrder);

// 生成幂等键，防止 MQ 重投后触发重复消费。
String idemKey = buildIdempotentKey(request);
```

## 5. 风格约束

- 优先使用中文注释；保留必要英文术语，如 DTO、MQ、idempotent、snapshot。
- 注释语气保持工程化、直接、可落地，避免写成需求文档或口语解释。
- 不改业务逻辑，只改注释和必要的格式整理；如果发现实现与注释冲突，要在结果中明确指出。
- 只有在复杂逻辑不加注释会明显影响可读性时，才补充多行说明。

## 6. 输出与交付要求

完成后按文件说明：

- 新增或更新了哪些类注释、方法注释、字段注释。
- 哪些方法补充了完整 JavaDoc。
- 哪些关键业务分支补充了行内注释。
- 哪些语义点仍需业务方确认。
