# 审查清单

审查新增代码时，将本文件作为详细规则参考。

## Elasticsearch

- 索引名只能使用小写字母、数字和安全分隔符，禁止大写字母和特殊字符。
- 文档写入必须使用 Bulk API，禁止在循环中逐条写入。
- 单批 Bulk 文档数不得超过 1000。
- 更新优先使用 `doc` 做部分更新，除非有充分理由，否则不要先查再全量覆盖。
- 禁止使用以 `*` 或 `?` 开头的通配符查询。
- 禁止使用 `from + size > 10000` 的深分页，必须改为 `search_after`、`scroll` 或 PIT + `search_after`。
- 必须使用 `_source` 过滤，仅返回必要字段。
- 必须显式设置 `size` 上限，默认不应超过 10000。
- 禁止对订单号、运单号、工单号、用户 ID 等高基数字段做 `terms` 聚合。
- `terms` 聚合必须显式设置 `size`，通常不超过 1000。
- 禁止在查询条件中使用脚本查询。
- 当可以明确指定索引时，禁止使用 `index_*` 这类宽泛索引模式查询。
- 大数据量导出必须使用 Scroll API 或 PIT + Search After，结束后必须清理游标。
- 精确过滤场景优先使用 `filter` 上下文，而不是依赖打分查询。

## 数据库与 ORM

- 禁止字符串拼接 SQL、`${}`、不安全的 `Wrapper.last(...)`，以及由外部输入直接拼装的动态 `order by`。
- 必须使用 `#{}` 或预编译参数；排序字段和方向必须走白名单或枚举映射。
- 对 `getOne`、`selectOne`、`one`、`findFirst` 或 `limit 1`，必须确认查询条件天然唯一，禁止用“取第一条”掩盖脏数据。
- 单条查询和分页查询都必须指定稳定的 `order by`。
- 禁止 `select *`，必须按需查询字段，尤其是分页、联表、存在性判断和大表场景。
- 仅做存在性判断时，优先使用 `exists`、`count`、`limit 1` 或最小字段查询，不要直接加载完整实体或列表。
- 无分页列表查询、导出查询、任务扫描和批处理扫描必须限制结果量或分批处理。
- 禁止默认使用大 offset 的深分页；大数据量场景必须改为游标分页、主键翻页或其他有界方案。
- 禁止在循环内逐条查库，必须先批量查询，再在内存中组装。
- `IN` 查询必须做空集合保护。
- 超大 `IN` 集合必须分批，避免 SQL 过长和执行计划劣化。
- 查询条件必须保持索引友好，禁止在索引列上做函数、表达式、类型转换或类型不一致比较。
- 按天或按小时查询时，优先使用左闭右开区间，例如 `>= start and < nextStart`，不要依赖 `23:59:59`。
- 复杂 join 和 XML SQL 必须审查过滤条件、字段宽度和结果集规模，避免弱条件大宽表联查。
- `count(*)` 或 `count(1)` 搭配 `join`、`group by`、`distinct` 时，必须确认统计口径和业务语义一致。
- 可以下推到 SQL 的筛选、排序、分页逻辑，禁止先查大列表再用内存 `stream().filter/sort/skip/limit` 处理。

## 批量、导入与导出边界

- 批量 List 入参必须限制大小，通常不超过 1000。
- 文件导入必须检查文件大小，超阈值时应拒绝或转为异步处理。
- 批量删除或更新必须带 `WHERE` 条件，禁止无条件全表操作。

## 并发与锁

- 高并发更新路径在业务需要时必须使用分布式锁。
- 获取多个锁时必须遵循全局一致顺序，避免死锁。

## 事务

- 事务中禁止包含 RPC、HTTP、消息发送等长时间阻塞操作。
- `@Transactional` 应显式指定 `rollbackFor = Exception.class`。
- 事务方法必须是 `public`，并关注自调用问题，因为代理事务不会在内部自调用时生效。
- 禁止在事务中执行大量非 DB 工作或大规模循环更新，应改为分批提交或缩小事务窗口。

## 空值与边界安全

- 参数和返回值在使用前必须判空，除非注解或接口契约已明确约束不可为空。
- 链式调用必须逐步检查空值安全。
- 除法或取模前必须检查除数不为 0。
- 通过下标访问集合或数组前必须检查边界。

## 异常处理

- 禁止空 `catch`，也禁止捕获 `Exception` 后既不处理也不记录日志。
- 禁止 `e.printStackTrace()`，必须使用结构化日志。
- 避免在 `finally` 中抛出新异常，以免掩盖原始异常。

## 日志规范

- 禁止日志输出密码、Token、证件号、银行卡号等敏感信息。
- 优先使用 `{}` 占位符，禁止字符串拼接日志。
- 异常日志必须包含堆栈，例如 `log.error("msg", e)`。

## 性能与资源使用

- 禁止在可批量或可缓存的场景下，在循环中反复发起 RPC、数据库查询或其他远程访问。
- 流、连接和结果集必须使用 `try-with-resources` 或等价方式安全关闭。
- 循环中的大字符串拼接优先使用 `StringBuilder`。
- 避免在循环中反复创建大对象。

## 可读性与可维护性

- 方法长度超过约 80 行时，应考虑拆分提取。
- 参数超过 5 个时，应考虑封装为对象。
- 嵌套超过 5 层时，应考虑扁平化或提取方法。
- 相同逻辑出现 2 处及以上时，应考虑收敛复用。
- 复杂逻辑应补充简洁注释，说明业务意图或非显而易见的约束。
