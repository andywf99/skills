# 代码规则审查

## 概述

针对生成或改动的代码执行规则审查。优先识别明确违规点、行为回归和缺失保护，不要把重点放在泛化的风格建议上。

## 审查流程

1. 明确审查范围。
只检查生成代码、改动代码，或用户明确指定的 diff，不要扩大到无关的历史遗留清理。

2. 加载规则参考。
阅读 [references/rules.md](./references/rules.md)，重点关注与改动相关的章节，例如灰度开关、Elasticsearch、RocketMQ、数据库访问、Redis、事务边界、并发、批处理、空值安全、异常、日志和资源使用。

3. 按风险而不是按文件顺序审查。
重点寻找：
- 灰度开关缺失或不规范
- 数据正确性风险
- 性能与资源风险
- 安全与合规违规
- 并发与事务隐患
- 边界控制缺失

4. 输出有证据支撑的问题。
每个问题都要指出具体代码路径或语句，并说明会怎样出错，不要只复述规则本身。

5. 以 findings 为主。
如果存在问题，先按严重级别列出 findings；如果未发现问题，也要明确说明，并补充剩余风险或未验证区域。

## 输出格式

使用代码审查格式输出。

- 先输出 findings。
- 每条 finding 都要包含严重级别、违反的规则、影响路径以及影响原因。
- 如果修复方式很明确，给出最小安全修复建议。
- findings 之后可补充开放问题或验证缺口。
- 总结保持简短。

## 严重级别参考

- High：可能导致全表或全索引扫描、数据损坏、死锁、缓存不一致、内存暴涨、敏感信息暴露或明显的生产稳定性问题。
- Medium：可能导致结果不稳定、额外负载、分页错误、边界检查缺失或持续性的性能退化。
- Low：主要是可读性弱、复杂逻辑缺少必要注释，或暂时不直接影响正确性与稳定性的可维护性风险。

## 重点检查项

- 灰度开关：历史代码修改的灰度开关覆盖、方法级和代码块级灰度开关使用规范、中台项目新接口的灰度开关。
- Elasticsearch：批量写入约束、查询范围、分页方式、字段过滤、聚合限制、scroll 使用规范和禁用模式。
- RocketMQ：循环发送、幂等性、异常处理、阻塞操作、Topic 命名、序列化格式。
- 数据库：参数安全、唯一性假设、稳定排序、字段收敛、结果量控制、循环查库、深分页和索引友好谓词。
- Redis：Key 命名、过期时间、大 Key 风险、锁安全、原子更新和事务一致性。
- 事务与锁：回滚配置、自调用问题、事务内远程调用、长临界区和锁顺序。
- 批量与导入导出：输入大小限制、文件大小限制、分片或分批策略，以及带条件的删除或更新。
- 通用代码安全：空值处理、异常日志、资源关闭、循环内高开销操作和多线程异常处理。

## 常见漏审点

- `getOne`、`selectOne`、`findFirst` 或 `limit 1` 在未证明唯一性或稳定排序的情况下直接取第一条。
- 只需要少量字段时仍然使用 `select *` 或全量文档返回。
- 查询或导出链路没有结果量上限。
- 在循环体中重复调用 mapper、repository、service、RPC 或 Redis。
- 事务块中包含 HTTP、RPC 或消息发送。
- Redis 锁缺少过期时间、锁归属校验或 `try/finally` 释放。
- Elasticsearch 查询使用前导通配符、深分页、宽泛索引模式、脚本查询或高基数聚合。
