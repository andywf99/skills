# Preview 转 TRD Skill

将已预审的需求文档（Markdown格式）转换为 TRD（概要设计文档）的技能，遵循极兔速递的概要设计规范。

---

## 适用场景

本 skill 适用于以下场景：
- 需求文档已经通过格式转换和需求评审（由 prd2trd skill 的前两步完成）
- 用户已有 Markdown 格式的需求文档，并已通过评审确认
- 需要直接进入需求深挖和 TRD 规划阶段

---

## 文件夹结构约定

每个需求对应一个文件夹，所有产出物集中存放：

```
#12xxxxx 【业务需求】XXX/
├── assets/                          ← 需求文档中的图片
├── 需求文档.md                      ← 已预审的markdown需求文档（Claude读这个）
├── research.md                      ← 第一步：需求深挖
├── plan.md                          ← 第二步：TRD规划
└── 概要设计.md                      ← 最终交付物
```

---

## 工作流程

```
需求深挖(Research) → TRD规划(Planning) → 批注迭代 → 生成TRD
```

### Step 1: 需求深挖(Research)

**前提**：用户提供已预审的需求文档（Markdown格式）

**目的**：深入分析需求，补充技术细节

**输出**：`research.md`

**分析内容**：
1. 功能点拆解
2. 业务规则梳理
3. 数据模型分析
4. 接口依赖梳理
5. 技术风险识别

**写完后停下来，等用户审阅并确认。**

---

### Step 2: TRD规划(Planning)

**前提**：用户确认 `research.md` 通过

**目的**：按照概要设计规范规划 TRD 结构

**输出**：`plan.md`

**规划内容**：
- 各章节的内容大纲
- 需要补充的技术细节
- 待确认的技术决策

**写完后停下来，等用户审阅并确认。**

---

### Step 3: 批注迭代

用户在 `research.md` 或 `plan.md` 中通过批注标记反馈：

| 标记 | 用途 | 示例 |
|------|------|------|
| `> **[批注]**` | 纠正事实、补充信息 | `> **[批注]** 这里少了代理区账号的越权场景` |
| `> **[否决]**` | 否定某个方案 | `> **[否决]** 这个方案不行，性能有问题` |
| `> **[补充]**` | 添加约束条件 | `> **[补充]** 还需要一个前置条件：网点必须是已激活状态` |
| `> **[疑问]**` | 提出问题 | `> **[疑问]** 这里的并发量预估是多少？` |
| `> **[确认]**` | 表示通过审阅 | `> **[确认]** research没问题，可以出plan了` |

**处理规则**：
- 读到 `[批注]`/`[否决]`/`[补充]`：修改对应内容，更新文档
- 读到 `[疑问]`：在文档中回答或直接回复用户
- 读到 `[确认]`：进入下一阶段
- **没有读到 `[确认]`，绝不进入下一阶段**

---

### Step 4: 生成TRD

**前提**：用户确认 `plan.md` 通过

**输出**：`概要设计.md`

**遵循极兔概要设计规范**，见下方模板。

## 极兔速递背景知识

### 一、组织架构

#### 1.1 组织层级结构

```
总部（最高层级）
  ├── 代理区
  │   ├── 虚拟代理区（下辖多个真实代理区）
  │   │   ├── 浙江代理区（杭州、宁波、温台、金华）
  │   │   ├── 广东代理区（广州、深圳、东莞、粤西、粤东）
  │   │   ├── 京津冀代理区（河北、天津、北京）
  │   │   ├── 甘青宁代理区（甘肃、青海、宁夏）
  │   │   ├── 江苏代理区（苏北、苏南）
  │   │   ├── 渝贵代理区（重庆、贵州）
  │   │   └── 东北代理区（吉林、黑龙江、辽宁）
  │   │
  │   └── 真实代理区（本身即独立代理区，无上级虚拟代理区）
  │       ├── 上海代理区、福建代理区、广西代理区、云南代理区
  │       ├── 新疆代理区、安徽代理区、山东代理区、河南代理区
  │       ├── 山西代理区、湖南代理区、川藏代理区、江西代理区
  │       ├── 陕西代理区、海南代理区、湖北代理区、内蒙古代理区
  │       └── 西藏代理区
  │
  ├── 转运中心（中心层级）
  │   └── 例如：上海浦西转运中心
  │
  ├── 加盟商（中心层级）
  │   └── 例如：上海加盟商三百七十七
  │
  └── 网点（最底层级）
      └── 例如：上海北青公路网点
```

#### 1.2 层级说明

| 层级           | 说明                                         | 示例                                             |
| -------------- | -------------------------------------------- | ------------------------------------------------ |
| **总部**       | 最高管理层级，负责全国业务统筹               | 极兔速递总部                                     |
| **虚拟代理区** | 多个真实代理区合并管理的大区                 | 浙江代理区（下辖杭州、宁波、温台、金华代理区）   |
| **真实代理区** | 实际运营的代理区，可独立或归属虚拟代理区     | 杭州代理区（属于浙江虚拟代理区）、上海代理区（独立） |
| **转运中心**   | 货物中转处理中心，属于中心层级               | 上海浦西转运中心                                 |
| **加盟商**     | 加盟合作伙伴，属于中心层级                   | 上海加盟商三百七十七                             |
| **网点**       | 最基层的服务网点                             | 上海北青公路网点                                 |

**注意**：
- 转运中心和加盟商都属于"中心层级"，在系统中具有相同的层级权限。
- 虚拟代理区括号内的名称即为其下辖的真实代理区（如"浙江代理区（杭州、宁波、温台、金华）"表示杭州代理区、宁波代理区、温台代理区、金华代理区归属浙江虚拟代理区管理）。
- 没有括号的代理区本身就是独立的真实代理区（如上海代理区、福建代理区）。

### 三、业务术语和概念

#### 3.1 核心业务模块

根据工作文档，主要涉及以下业务模块：

1. **无头件管理**
   - 无头件登记、认领、审核
   - 无头件内物信息查询
   - 残单检索
   - 作废件处理

2. **问题件管理**
   - 问题件登记、回复、查询
   - 问题件转作废、转留仓
   - 问题件监控

3. **工单系统**
   - **仲裁工单**：处理快件纠纷和判责
   - **质量工单**：处理质量问题
   - **监控工单**：处理监控发现的问题
   - **平台工单**：处理平台相关工单

4. **完结件管理**
   - 完结件记录生成
   - 完结网点取值逻辑
   - 不同来源数据进入完结件

5. **作废件管理**
   - 无头件自动转作废
   - 作废件审核流程
   - 作废件处理

#### 3.2 常见业务术语

| 术语             | 说明                                                   |
| ---------------- | ------------------------------------------------------ |
| **无头件**       | 有实物但没有运单（面单丢失/损坏）的包裹                |
| **残单检索**     | 通过描述物品信息（收寄件人、地址等）查找运单号         |
| **内物信息查询** | 通过运单号查询包裹内物品信息（商品名称、照片、价值等） |
| **作废件**       | 无法继续派送需要作废处理的快件                         |
| **完结件**       | 记录快件异常完结状态的记录                             |
| **问题件**       | 在运输过程中出现问题的快件                             |
| **仲裁工单**     | 处理快件纠纷和责任判定的工单                           |
| **质量工单**     | 处理质量问题的工单                                     |
| **监控工单**     | 通过监控发现问题的工单                                 |
| **平台工单**     | 来自电商平台的工单                                     |
| **代理区转责**   | 代理区重新划分责任网点的操作                           |
| **认领**         | 网点认领无头件的过程                                   |
| **审核**         | 对认领进行审核确认的过程                               |

#### 3.3 数据来源说明

在无头件内物查询中，数据来源包括：

- **菜鸟接口**：外部平台接口
- **平台工单**：极兔内部平台工单系统
- **仲裁工单**：极兔内部仲裁工单系统
- **客户物品信息库**：人工导入的客户物品信息

概要设计归档目录：[概要设计](https://jtexpress.feishu.cn/wiki/Q34YwJXgBi2sXXkHiweck8vLn7c)

### 1. 需求背景

| |

### 2. 业务链路

|                                                             |
|-------------------------------------------------------------|
| 引入项目文档中的业务流程图/泳道图/时序图，并加以修改（流程改动点用不同颜色标注出来）,可以使用mermaid格式的 |

### 3. 数据结构

**要求**：
- 数据库表结构：定义 E-R 模型，输出 DDL 语句（合理设置表字段长度上限、索引）
- 数据幂等设计：所有表都要保证有唯一索引，防止并发导致重复数据入库
- 特别注意：工单类、订单类、配置类、流水类、短信类业务必须有唯一索引
- 无法增加唯一索引的业务需求，必须提供充足的理由并审核通过

**输出内容**：
- 表结构设计
- DDL语句
- 索引设计
- 幂等方案

### 4. 数据回刷

参考：[服务质量-刷数据影响范围评估规范](https://jtexpress.feishu.cn/wiki/QqpQwBgO7iLTVVkiO0PcO8H9n1d)

**输出内容**：
- 回刷场景说明
- 影响范围评估
- 回刷脚本
- 回滚方案

### 5. 接口设计

**要求**：
- 接口文档：接口url、入参、出参
- 并发设计：新增、变更类必须要有分布式乐观锁/悲观锁
- 事务设计：多表新增或更新场景必须增加事务
- 日志设计：新增、变更类必须要有详细日志留痕

**输出内容**：
- 接口列表
- 接口详细设计（URL、方法、入参、出参、错误码）
- 并发控制方案
- 事务边界
- 日志埋点

### 6. 性能设计

**要求**：
- 接口压测：所有接口必须在需求评审阶段确认性能指标
- 外域接口：提供给第三方、外域的接口必须经过平台压测，上线后使用 Sentinel 限流
- 性能考虑：确认是否会出现慢查询？是否存在 OOM？
- 业务并发量大的读写场景，必须汇报给旗主进行方案及压测结果确认
- 数据限制：导入、导出、批处理接口是否有数量限制？

**输出内容**：
- 性能指标要求
- 压测方案
- 慢查询分析
- OOM 风险评估
- 分表/ES/ADB 等中间件使用方案

### 7. 容错设计

**要求**：
- 异步异常预警：异步/JOB 中必须对代码块进行异常捕获，并在异常时飞书告警
- 数据增量预警：数据量突增或超出阈值时需告警
- 避免批处理阻塞：批量处理时，单条数据出现已知异常需跳过并告警，不可阻塞后续处理

**输出内容**：
- 异常处理方案
- 告警机制
- 批处理容错方案

### 8. 灰度设计

参考：[灰度发布](https://jtexpress.feishu.cn/wiki/YG8SwLwnmiqotgkE0O1cAyPgnNg)

**输出内容**：
- 灰度发布方案
- 回滚方案
- 灰度开关配置

### 9. 安全设计

**要求**：
- 权限校验：需保证无水平和垂直越权
- 数据脱敏：包含但不限于手机号、身份证号、银行卡号、收寄件地址、邮箱

**输出内容**：
- 权限设计
- 脱敏方案
- 安全风险评估

## 执行检查清单

在生成 TRD 前，确认以下内容：

- [ ] 需求背景是否清晰？
- [ ] 业务链路是否完整？
- [ ] 数据结构是否有唯一索引保证幂等？
- [ ] 接口设计是否包含并发控制和事务设计？
- [ ] 性能设计是否明确指标和压测方案？
- [ ] 容错设计是否覆盖异步/JOB场景？
- [ ] 灰度设计是否有回滚方案？
- [ ] 安全设计是否包含权限和脱敏？
- [ ] 发版信息是否完整？

## 注意事项

1. **不要跳过流程**：必须按顺序执行 Research → Planning，每个阶段都要等用户确认
2. **保持文档一致性**：各文档之间要保持逻辑一致，发现矛盾及时指出
3. **关注细节**：概要设计要具体可执行，不要留空或写模糊描述
4. **遵循规范**：严格按照极兔的概要设计规范输出
5. **图片处理**：检查需求文档中的图片路径是否正确
