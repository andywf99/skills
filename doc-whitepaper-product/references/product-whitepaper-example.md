# 仲裁中台产品白皮书

## 1. 产品概述

### 1.1 产品定位

仲裁中台是极兔物流供应链管理系统的核心组成部分，是物流异常处理与责任认定的专业化平台。系统以运单为载体，围绕延误、破损、遗失、违规件等异常场景，提供从申报、处理、审核、申诉到结案的全生命周期管理能力。

核心定位：
- **物流异常处理中心**：统一处理全网物流运作中产生的各类异常问题
- **责任认定与裁定平台**：通过规范化流程确定责任归属和赔偿金额
- **数据驱动的智能决策系统**：利用AI技术辅助智能裁定，提升处理效率
- **财务结算枢纽**：与财务系统对接，实现赔偿款项的自动结算

### 1.2 目标用户画像

| 用户角色 | 所属组织 | 核心诉求 | 使用场景 |
|---------|---------|---------|---------|
| 申报网点客服 | 网点/加盟商 | 快速申报异常、追踪处理进度 | 面对客户投诉时发起仲裁申报 |
| 责任网点负责人 | 网点/加盟商 | 查看被投诉信息、进行认责/申诉 | 接收仲裁通知、提交证据材料 |
| 仲裁员 | 总部/代理区 | 审核材料、作出裁定决策 | 分配案件、核实情况、裁定金额 |
| 网点经营管理人员 | 代理区/总部 | 查看网点仲裁数据、分析运营质量 | 数据分析、绩效考核 |
| 财务人员 | 总部/代理区 | 处理赔偿结算、生成财务报表 | 审核流水、对接财务系统 |

### 1.3 核心价值主张

1. **规范化异常处理流程**：建立标准的仲裁申报、处理、审核、申诉、结案流程，确保处理过程透明、公正
2. **提升处理效率**：通过智能裁定、自动抓取延误数据、自动化任务等手段，大幅降低人工处理成本
3. **数据资产沉淀**：积累历史仲裁数据，为网点考核、运营优化提供数据支撑
4. **财务闭环管理**：实现仲裁赔偿与财务系统的自动对接，确保账务准确

## 3. 业务规则汇编

### 3.1 时间配置规则

系统通过 `ArbitrationTime` 配置关键时间节点：

| 时间类型 | 说明 | 业务影响 |
|---------|------|---------|
| 转仲裁时间 | 责任方处理阶段自动转仲裁的时限 | 超时后自动转为待分配状态 |
| 申诉剩余时间 | 仲裁裁定完成后允许申诉的时间窗口 | 超时后自动结案 |
| 转撤销时间 | 申报撤销处理的时限 | 影响申报状态流转 |

### 3.2 自动裁定规则

**自动延误抓取规则**：
- 派送延误：单责/多责两种模式，支持剔除数据
- 中转延误：按时效规则自动判定责任方
- 集散延误：集散中心延误责任认定
- 退转件延误：退转件处理延误
- 绕行延误：路由绕行导致的延误
- 发件延误：发件环节延误
- 超7天未签收延误：长时间未签收自动创建
- 正班作废延误：班次作废导致的延误

**自动结案规则**：
- 仲裁裁定完成后，按申诉剩余时间配置自动结案
- 支持按一级类型配置不同的结案时间
- 分布式锁保证任务执行的唯一性

### 3.3 裁定金额计算

通过 `ArbitrationFee` 配置手续费规则：

| 计费方式 | 说明 | 适用场景 |
|---------|------|---------|
| 固定金额 | 按固定金额收取手续费 | 标准化异常类型 |
| 百分比 | 按物品价值百分比收取 | 高价值物品 |
| 按物品价值裁定 | 基于物品价值确定裁定金额上限 | 遗失/破损赔偿 |
| 延误扣款 | 按延误天数和扣款金额计算 | 延误类仲裁 |

### 3.4 异常类型分类

**一级异常类型**（来自 `ArbitrationTypeFirstCodeEnum`）：

| 编码 | 名称 | 说明 |
|-----|------|------|
| 01 | 延误 | 派送/中转/集散等各类延误 |
| 02 | 破损 | 快件包装或内件损坏 |
| 03 | 遗失 | 快件丢失 |
| 04 | 总部极速理赔 | 总部层面的快速理赔通道 |
| 05 | 代理区极速理赔 | 代理区层面的快速理赔通道 |
| 06 | 违规件 | 违反运输规定的快件 |
| 07 | 漏录签收 | 签收信息未录入系统 |
| 08 | 农特专递 | 农产品特产专递相关 |
| 15 | 春节不打烊 | 春节期间服务保障 |
| 17 | 疑似遗失 | 疑似丢失但未确认 |
| 25 | 自动抓取 | 系统自动抓取的异常 |
| 26 | 虚假问题件 | 虚假申报的问题件 |
| 50 | 项目考核 | 项目层面的考核 |

**二级异常类型（关键）**：
- 延误细分为：派送延误、中转延误、集散延误等
- 破损细分为：外包装破损、内件破损、部分破损等
- 遗失细分为：全程遗失、包内遗失等

## 5. 核心功能详解

### 5.1 仲裁申报管理

#### 5.1.1 申报入口

系统支持多种申报入口，满足不同业务场景需求：

| 申报方式 | 入口系统 | 来源编码 | 适用场景 | 触发条件 |
|---------|---------|---------|---------|---------|
| 人工申报 | 服务质量平台 | 81 | 客户投诉、网点纠纷 | 客服手动发起 |
| 批量导入 | 服务质量平台 | 83 | 批量异常件处理 | Excel模板导入 |
| 理赔转仲裁 | 理赔系统 | 82 | 理赔申请转仲裁 | 理赔审核通过后转仲裁 |
| 工单转仲裁 | 工单系统 | 103/104 | 工单认责转仲裁 | 工单认责确认后转仲裁 |
| 疑似遗失申报 | 操作平台 | 21 | 轨迹断更异常 | 超过滞留天数自动推送 |
| 网点考核导入 | 网点经营 | 51 | 考核数据导入 | 项目考核需要 |
| 延误自动抓取 | 经营指标 | 91-99 | 各类延误时效 | 时效系统自动判定 |

#### 5.1.2 申报必填信息

**基础信息**：
- 运单号（waybillNo）：必填，最大30字符
- 物品名称（goodsName）：必填，最大60字符
- 物品价值（goodsValue）：必填，用于裁定金额计算
- 物品类型（goodsTypeId/code/name）：选填

**异常信息**：
- 一级类型（firstTypeId）：必填，延误/破损/遗失等
- 二级类型（secondTypeId）：必填，细分异常类型
- 三级备注（thirdRemark）：选填
- 异常说明（exceptionDesc）：必填，最大500字符
- 延误天数（delayDay）：延误类必填，≥1

**网点信息**：
- 投诉网点（complaintNetworks）：必填，支持多网点
  - 网点ID（networkId）
  - 网点编号（networkCode）
  - 网点名称（networkName）
  - 所属代理区（parentId/code/name）

**附件信息**：
- 图片/文件路径（paths）：选填，支持多文件上传

#### 5.1.3 申报校验规则

1. **运单校验**：
   - 运单号格式校验
   - 运单是否存在检查
   - 是否已有同类型仲裁单检查

2. **网点校验**：
   - 投诉网点不能为共享网点
   - 投诉网点不能重复
   - 网点必须存在且有效
   - 虚拟代理区不能创建仲裁

3. **类型校验**：
   - 异常类型必须启用
   - 延误类必须填写延误天数
   - 遗失类自动判断是否包内遗失

#### 5.1.4 申报后状态

| 场景 | 初始状态 | 后续流转 |
|-----|---------|---------|
| 标准申报 | 申报方处理中(12) | 提交后→责任方申诉举证(1) |
| 智能破损认责 | 申报方填写物品价值(21) | 填写后→责任方确认物品价值(22) |
| 疑似遗失 | 申报处理-疑似遗失(20) | 认责后→正常流程 |

### 5.2 申报方处理阶段

#### 5.2.1 阶段说明

申报方处理阶段（CREATE_PROCESSING, 状态12）是申报创建后的初始处理阶段，主要用于：
- 申报信息补充完善
- 申报材料准备
- 申报确认提交

#### 5.2.2 可执行操作

| 操作 | 接口 | 说明 | 状态变更 |
|-----|-----|------|---------|
| 编辑 | /arbitration/edit | 修改申报信息 | 保持不变 |
| 提交仲裁 | /arbitration/createProcessSubmitArb | 确认提交 | →责任方申诉举证(1) |
| 撤销 | /arbitration/createProcessRevoke | 撤销申报 | →已撤销(2) |

#### 5.2.3 智能破损认责特殊流程

```
申报方填写物品价值(21)
        │
        ▼
责任方确认物品价值(22)
        │
        ├── 确认价值 ──→ 自动裁定
        │
        └── 异议 ──→ 转标准流程
```

---

### 5.3 责任方处理阶段

#### 5.3.1 阶段说明

责任方处理阶段（RESP_PROCESSING, 状态1）是责任网点对仲裁申报进行回应的阶段。

#### 5.3.2 处理时限

系统通过`ArbitrationTime`配置转仲裁时间，超时后自动转为待分配状态。

#### 5.3.3 可执行操作

| 操作 | 说明 | 状态变更 | 业务影响 |
|-----|------|---------|---------|
| 责任方回复 | 提交证据材料 | 保持不变 | 回复状态变为"已回复" |
| 责任方认责 | 确认承担责任 | 保持不变 | 认责状态变为"已认责" |
| 责任方申诉 | 对申报有异议 | 保持不变 | 需仲裁员介入裁定 |

#### 5.3.4 责任方回复内容

- 回复意见（processOpinion）：最大500字符
- 附件材料（paths）：图片、证明文件等
- 回复时间记录

#### 5.3.5 自动流转规则

```
责任方申诉举证(1)
        │
        ├── 超时未处理 ──→ 待仲裁分配(3)
        │
        ├── 申报方撤销 ──→ 已撤销(2)
        │
        └── 责任方全部认责 ──→ 可快速裁定
```

---

### 5.4 仲裁审核阶段

#### 5.4.1 案件分配

**分配方式**：

| 方式 | 说明 | 触发条件 | 接口 |
|-----|------|---------|------|
| 人工分配 | 管理员指定处理人 | 待分配列表选择 | /arbitration/assign |
| 自动分配 | 系统按规则分配 | 定时任务扫描 | 定时任务 |
| 申领模式 | 仲裁员主动领取 | 点击申领按钮 | /arbitration/apply |

**分配参数**：
- 处理人ID（assignProcessUserId）
- 处理人编号（assignProcessUserCode）
- 处理人名称（assignProcessUserName）
- 分配类型（assignType）：仲裁/申诉

**分配后状态**：
- 待仲裁分配(3) → 仲裁受理中(4)

#### 5.4.2 仲裁核实

**核实阶段**（ARB_VERIFYING, 状态5）：

仲裁员查看申报材料和责任方回复后，可发起核实：

| 操作 | 说明 | 状态变更 |
|-----|------|---------|
| 发起核实 | 需要更多信息 | →仲裁核实中(5) |
| 核实回复 | 补充材料回复 | 保持不变 |
| 核实完成 | 核实结束 | →仲裁受理中(4) |

**核实参数**：
- 责任网点集合（responsibilityNetworks）：必填
- 核实意见（processOpinion）
- 附件材料（paths）

#### 5.4.3 仲裁退回

**退回场景**：
- 材料不齐全
- 信息有误需要补充
- 需要申报方重新确认

**退回参数**：
- 退回意见（processOpinion）：必填，最大500字符
- 二次系统使用费（systemUsageFeeSecond）

**退回后状态**：
- 仲裁受理中(4) → 仲裁退回(6)
- 责任方申诉举证(1) → 待仲裁分配(3)

#### 5.4.4 仲裁裁定

**裁定阶段**是仲裁员作出最终判决的关键环节。

**裁定参数**（ArbitrationAdjudicateDTO）：

| 参数 | 必填 | 说明 |
|-----|------|------|
| arbitrationId | 是 | 仲裁单ID |
| list | 是 | 裁定项集合（支持多责任方） |
| firstTypeId | 是 | 一级类型ID |
| secondTypeId | 是 | 二级类型ID |
| delayDay | 延误类必填 | 延误天数 |
| isAppealFee | 否 | 是否有申诉费用 |
| isFirstError | 否 | 是否首次错误 |
| firstErrorDesc | 否 | 错误描述 |

**单条裁定项参数**（AdjudicationDTO）：

| 参数 | 说明 |
|-----|------|
| responsibilityNetworkId/Code/Name | 责任网点信息 |
| payableAmount | 应付金额 |
| handlingFee | 手续费 |
| settlementNetworkId/Code/Name | 财务结算网点 |

**裁定流程**：
```
仲裁受理中(4)
        │
        ▼
    仲裁裁定
        │
        ├── 选择责任网点
        │
        ├── 计算裁定金额
        │   ├── 物品价值 × 比例
        │   ├── 固定金额
        │   └── 延误扣款：天数 × 单价
        │
        ├── 确定手续费
        │
        └── 提交裁定 → 仲裁裁定完成(7)
```

#### 5.4.5 批量裁定

支持对同类案件进行批量裁定：
- 接口：/arbitration/batchAdjudicate
- 适用场景：延误类批量处理
- 校验：责任网点不能为共享网点

#### 5.4.6 自动裁定

系统支持自动裁定场景：
- 延误自动抓取类案件
- 责任方已认责案件
- 智能破损认责确认后

---

### 5.5 申诉处理阶段

#### 5.5.1 申诉发起

**发起条件**：
- 仲裁裁定完成(7)后
- 在申诉剩余时间内
- 责任网点有异议

**申诉参数**：
- 申诉意见
- 申诉材料
- 申诉网点信息

**申诉后状态**：
- 仲裁裁定完成(7) → 申诉受理中(8)

#### 5.5.2 申诉分配

与仲裁分配类似：
- 待分配状态：申诉待分配(11)
- 分配后状态：申诉受理中(8)

#### 5.5.3 申诉核实

| 操作 | 状态变更 |
|-----|---------|
| 发起核实 | 申诉受理中(8) → 申诉核实中(9) |
| 核实回复 | 保持不变 |
| 核实完成 | →申诉受理中(8) |

#### 5.5.4 申诉裁定

**裁定参数**与仲裁裁定类似，增加：
- 是否有申诉费用（isAppealFee）
- 申诉费用金额

**申诉裁定后**：
- 申诉核实中(9) → 已结案(10)

#### 5.5.5 申诉撤销

责任方可在申诉阶段撤销申诉：
- 接口：/arbitration/rescind
- 撤销后：恢复到仲裁裁定完成(7)

---

### 5.6 结案与后续处理

#### 5.6.1 自动结案

**触发条件**：
- 仲裁裁定完成后超过申诉剩余时间
- 未发起申诉

**定时任务**：`arbitrationCloseNewTask`

**处理逻辑**：
1. 从ES查询符合条件的仲裁单
2. 按一级类型分组处理
3. 分布式锁保证幂等
4. 生成财务流水
5. 更新状态为已结案(10)

#### 5.6.2 结案撤回

**条件校验**：
- 财务流水未推送
- 或流水可回退

**操作**：
- 接口：/arbitration/closureWithdrawal
- 状态：已结案(10) → 仲裁裁定完成(7)

#### 5.6.3 裁定撤销

仲裁员可撤销已作出的裁定：
- 接口：/arbitration/cancelArbitration
- 场景：裁定错误需要重新裁定
- 状态：仲裁裁定完成(7) → 仲裁受理中(4)

#### 5.6.4 重启仲裁

对于特殊案件可重启仲裁流程：
- 接口：/arbitration/batchRestart
- 重启次数记录（restartCount）

---

### 5.7 财务结算管理

#### 5.7.1 财务流水生成

**生成时机**：
| 场景 | 生成时间 | 流水类型 |
|-----|---------|---------|
| 仲裁裁定完成 | 结案时 | 裁定金额、应付金额、手续费 |
| 申报方撤销 | 撤销时 | 退款流水 |
| 申诉裁定 | 申诉裁定时 | 申诉费用 |

**流水类型**（costType）：
- 1：裁定金额
- 2：应付金额
- 3：系统使用费
- 4：申诉费用

**交易类型**（transactionType）：
- 1：加款
- 2：减款

#### 5.7.2 财务流水推送

**推送流程**：
```
财务流水生成
        │
        ▼
   推送至IBK财务系统
        │
        ├── 成功 → 更新推送状态(1)
        │
        └── 失败 → 记录失败原因，支持重推
```

**推送方式**：
- 自动推送：结案后自动触发
- 手动推送：/arbitration/financePush

#### 5.7.3 手续费计算

**计费方式**（ArbitrationFee）：
| 方式 | type | 说明 |
|-----|------|------|
| 固定金额 | 1 | 按配置的固定金额 |
| 百分比 | 2 | 物品价值 × 百分比 |

**延误扣款计算**：
```
扣款金额 = 延误天数 × 扣款单价/天
手续费 = 固定金额 或 物品价值 × 比例
```

#### 5.7.4 代理区结算

针对代理区层面的汇总结算：
- 按代理区维度汇总流水
- 生成代理区结算明细
- 支持结算状态跟踪

### 10.3 表关系说明

**主从关系**：

| 主表 | 从表 | 关系 | 关联字段 | 说明 |
|------|------|------|----------|------|
| css_arbitration | css_adjudication | 1:N | arbitration_id | 一个仲裁单可有多条裁定（多责任方） |
| css_arbitration | css_arbitration_network | 1:N | arbitration_id | 支持多网点投诉/多责任方 |
| css_arbitration | css_arbitration_process_record | 1:N | arbitration_id | 全流程留痕记录 |
| css_arbitration | css_arbitration_finance_record | 1:N | arbitration_id | 财务流水（裁定/应付/手续费/申诉费） |
| css_arbitration_type_first | css_arbitration_type_second | 1:N | first_id | 类型层级关系 |
| css_arbitration_type_first | css_arbitration_fee | 1:N | first_id | 手续费按类型配置 |
| css_arbitration_type_first | css_arbitration_deduction | 1:N | first_type_id | 扣款配置（主要用于延误类） |

**索引设计要点**：

1. **主表索引**：
   - `uk_waybill_no_first_type`：运单号+一级类型唯一索引（防重复申报）
   - `idx_status`：状态索引（待处理列表查询）
   - `idx_create_network_id`：申报网点索引
   - `idx_first_resp_network_id`：第一责任网点索引
   - `idx_create_time`：创建时间索引（时间范围查询）

2. **从表索引**：
   - `idx_arbitration_id`：仲裁单ID索引（关联查询）
   - `idx_arbitration_code`：仲裁单编号索引

3. **ES索引补充**：
   - `css_arbitration`：仲裁单搜索查询（支持组合条件）
   - `css_arbitration_network`：网点维度数据聚合

### 10.4 数据流转示意

```
仲裁申报创建
      │
      ├──► css_arbitration (主表插入)
      │
      ├──► css_arbitration_network (网点关系插入)
      │         ├── 投诉网点(type=1)
      │         └── 责任网点(type=2)
      │
      └──► css_arbitration_process_record (处理记录插入)
                └── process_type=1(仲裁申报)

责任方处理/仲裁审核
      │
      ├──► css_arbitration.status 更新
      │
      └──► css_arbitration_process_record (追加处理记录)
                ├── process_type=2(责任方回复)
                ├── process_type=5(仲裁分配)
                ├── process_type=6~8(核实/退回)
                └── process_type=9(仲裁裁定)

仲裁裁定
      │
      ├──► css_adjudication (裁定结果插入)
      │         ├── responsibility_network_id
      │         ├── payable_amount
      │         └── handling_fee
      │
      └──► css_arbitration.total 更新

结案/财务流水
      │
      ├──► css_arbitration.status = 10(已结案)
      │
      └──► css_arbitration_finance_record (财务流水插入)
                ├── cost_type=1(裁定金额)
                ├── cost_type=2(应付金额)
                ├── cost_type=3(系统使用费)
                └── transaction_type=1(加款)/2(减款)
```
