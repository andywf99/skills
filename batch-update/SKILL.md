# 批量代码变更

当用户说 **[批量代码变更]**、**[batch update]**，或任务明显属于“同一类改动需要落到多个仓库”时，按以下流程执行。

## 使用原则

1. skill 只固化流程，不固化业务目标
2. 当前任务的改动规则，以用户本次要求为准
3. 先抽样确认，再批量执行，避免整批误改
4. 默认做最小必要变更，不顺手改无关问题
5. 每个仓库独立校验、独立记录结果
6. 未被明确要求时，不自动推送远端、不自动合并目标分支

---

## 任务 1：读取批量输入

**目标**：确定需要处理的项目、仓库地址、分支和本次变更目标。

**优先读取的输入**：
1. 当前目录下的 `projects.txt`
2. 当前目录下的 `gits.txt`
3. 当前目录下的可选说明文件，例如：
   - `change-spec.md`
   - `batch-update.md`
   - `branches.txt`
   - `targets.txt`

**处理规则**：
1. 解析项目名称和 Git 地址，建立一一映射
2. 如果存在分支说明，读取目标分支、基线分支和提交要求
3. 如果存在变更说明文件，提取：
   - 变更目的
   - 目标文件或模块
   - 搜索关键字
   - 需要新增、替换、删除的内容
   - 验证方式
4. 如果没有独立说明文件，则直接以用户当前消息作为变更规格

**验证**：
1. 每个项目都应有对应的 Git 地址
2. 需要明确本次改动的成功标准
3. 对模糊项先形成执行假设，再进入抽样阶段验证

---

## 任务 2：准备仓库与工作分支

**目标**：让所有待处理仓库进入可修改状态。

**操作步骤**：
1. 在当前目录下创建 `projects` 目录存放仓库
2. 遍历项目列表，对每个项目执行：
   - 如果本地不存在，执行 `git clone <git_url>`
   - 如果本地已存在，先读取当前状态，再按需 `fetch/pull`
3. 按本次任务要求切换分支：
   - 如果用户指定了工作分支，则切到该分支
   - 如果未指定，则基于当前任务生成语义明确的分支名
4. 保留每个仓库的基线信息：
   - 当前分支
   - 最新提交
   - 是否存在未提交改动

**注意事项**：
1. 不要覆盖用户已有的本地修改
2. 如果仓库工作区不干净，先记录并评估是否可继续
3. 分支策略以用户要求和仓库约定为准，不写死为某个固定分支

---

## 任务 3：抽样分析并固化改法

**目标**：先在少量代表性项目中确认正确改法，再放量执行。

**操作步骤**：
1. 选择 1 到 2 个代表性项目作为样板
2. 在样板项目中完成以下分析：
   - 技术栈和构建方式
   - 目标代码所在位置
   - 需要改动的文件类型
   - 是否已有类似实现
   - 是否存在环境差异或版本差异
3. 用搜索工具定位关键入口，例如：
   - 依赖坐标
   - 配置项
   - 类名
   - 方法名
   - API 路径
   - SQL 或脚本关键字
4. 形成本次任务的可复用改法，包括：
   - 哪些文件要改
   - 改动模式是什么
   - 哪些情况跳过
   - 哪些情况需要人工确认
   - 要执行哪些验证命令

**输出要求**：
1. 将改法总结成统一规则，再用于全量项目
2. 如果发现仓库之间差异较大，按类型分组后分别执行
3. 如果抽样结果与最初要求冲突，以代码现实为准修正方案

---

## 任务 4：批量实施变更

**目标**：对全部目标仓库执行同类改动。

**通用做法**：
1. 先搜索，再修改，不凭猜测直接改
2. 优先复用已有模式，保持原项目代码风格和目录结构
3. 改动粒度尽量小，避免引入无关格式化噪音
4. 对已符合要求的项目，标记为 **无需变更**

**常见批量变更类型**：
1. 新增或替换依赖
2. 补充或调整配置项
3. 增加灰度开关或兼容开关
4. 批量替换接口、类名、字段名、枚举值
5. 注入统一的线程池、拦截器、过滤器、starter、SDK
6. 清理废弃逻辑并保留兼容兜底
7. 调整脚本、SQL、部署文件、流水线配置

**执行要求**：
1. 每个仓库完成修改后立即做一次本地自检
2. 只在确定无歧义时自动修复
3. 如果某个仓库结构明显不同，单独记录，不强行套模板

---

## 任务 5：验证变更结果

**目标**：确认批量改动没有破坏仓库结构和主要行为。

**验证层次**：
1. 文件级验证
   - 目标文本已新增、替换或删除
   - 文件语法结构未损坏
   - 配置、XML、JSON、YAML、SQL 格式完整
2. 仓库级验证
   - 根据技术栈执行最小必要校验，例如编译、单测、lint、启动检查
   - 如果全量执行成本过高，至少对样板项目执行完整校验
3. 结果级验证
   - 对照本次变更规格逐项确认达成

**注意事项**：
1. 优先执行成本低且能快速发现问题的验证
2. 如果受环境限制无法运行验证命令，必须在总结中明确说明
3. 验证失败的仓库不要混入成功列表

---

## 任务 6：按需提交、推送、合并

**目标**：根据本次任务要求处理 Git 动作。

**默认策略**：
1. 如果用户只要求“改代码”，则完成本地修改和验证即可
2. 如果用户明确要求提交，则执行 `git add` 和 `git commit`
3. 如果用户明确要求推送，则执行 `git push`
4. 如果用户明确要求合并目标分支，则再执行 `merge` 或 `rebase`

**执行要求**：
1. 提交信息应准确描述本次批量变更主题
2. 推送前再次确认当前分支正确
3. 遇到冲突时先分析来源，再决定是否自动处理
4. 无法安全自动解决的冲突，保留现场并记录到失败列表

---

## 任务 7：输出批量执行报告

**目标**：给出可直接复核的执行结果。

**最终报告至少包含**：

### 变更成功的项目

| 项目名称 | Git 地址 | 分支 | 结果 | 验证情况 |
|---------|---------|------|------|---------|
| project-a | git@xxx | feature/batch-update | 已完成批量变更 | 编译通过 |

### 无需变更的项目

| 项目名称 | Git 地址 | 原因 |
|---------|---------|------|
| project-b | git@xxx | 已符合本次要求 |

### 失败项目

| 项目名称 | Git 地址 | 失败阶段 | 失败原因 |
|---------|---------|---------|---------|
| project-c | git@xxx | 验证 | 测试失败，需人工排查 |

### 补充说明

1. 本次批量变更的统一规则摘要
2. 实际执行过的验证命令
3. 未执行的验证及原因
4. 需要人工继续处理的仓库和建议动作

---

## 执行偏好

1. 优先用搜索结果驱动修改，不凭文件名猜结构
2. 优先对样板项目做透彻，再复制到同类项目
3. 优先保留兼容性，除非用户明确要求清理旧逻辑
4. 优先输出结构化结果，便于后续复盘和补跑
